## Donation-flow — текущий поток монетизации (✅ 2026-05-27, упрощённый курс)

Владелец 2026-05-27 (вечер): «устал от бесконечной борьбы с провайдерами, делаем простую donation-схему». Стало курсом по умолчанию.

**Как работает (end-to-end):**
1. Юзер в ЛК / в боте видит реквизиты Т-Банк (СБП `+79213032918` или карта `2200 7007 6046 4759`) + кнопку **«✅ Я перевёл деньги, подтверди»**.
2. Жмёт → создаётся `payment_claim` (`status='pending'`) → владельцу в TG приходит сообщение с инлайн-кнопками «✅ Подтвердить +30 дн» / «❌ Отклонить».
3. Владелец проверяет в банке поступление, жмёт «Подтвердить» → бот зачисляет +30 дней через `db_extend_subscription` (стакается с остатком), записывает платёж как `provider='manual'` + `external_id='claim:<id>'` (идемпотентно), применяет реферал-бонус, шлёт юзеру «✅ Подписка продлена до DATE».
4. Если «Отклонить» → юзер получает «Оплата не подтверждена, проверь перевод и нажми ещё раз».
5. **Один pending на юзера** — повторные нажатия только переотправляют уведомление (не плодят дубли).

**Файлы:**
- БД: таблица `payment_claims` + флаги `notif_7d/3d/0d_sent` в `users`.
- Web: `POST /api/billing/claim-payment` (email-auth) → создаёт claim + дёргает TG sendMessage.
- Bot: callback handlers `claim_approve:` / `claim_decline:`, кнопка «💳 Продлить подписку» в главном меню, экран реквизитов с кнопкой «Я перевёл».
- ЛК: пейBlock переключается в pending-режим если `d.pending_claim`.

**Cron-напоминания (✅ 2026-05-27):**
- `scripts/expiry_reminder.py` крутится через cron `0 9 * * *` (12:00 МСК) на Fornex.
- Шлёт юзерам сообщения T-7 / T-3 / T-0 до окончания подписки (точная календарная дата). Идемпотентно через флаги `notif_*_sent`, которые сбрасываются при каждом продлении.
- Grandfather (`expires_at IS NULL`) пропускаются.
- Если юзер заблокировал бота (403) — флаг ставится всё равно (не ретраим).

**Что НЕ требует enforcement.** Текущие 34 grandfather-юзера остаются с `expires_at = NULL` — для них напоминаний не будет, доступ безграничный. Как только первый платит — у него ставится `expires_at = now + 30 дн` и дальше пошла обычная подписка. Это мягкий вход.

**Stars-канал остаётся** (commit `23cf3c5` от утра): две кнопки в ЛК «⭐ Подписка Stars (150⭐/мес, авто)» и «⭐ Оплатить один раз». Авто-зачисление через `successful_payment`-handler, без участия владельца. Не мешают и не дублируют donation-flow — это два параллельных канала.

**`/admin/credit` остаётся** как fallback на случай если бот упадёт или inline-кнопки отвалятся.

- **MTProxy — revival после волны РКН** (2026-06-05):
  - MTProxy fake-TLS задавлен ТСПУ на уровне протокола: исчерпали домен (`1c.ru`→`swcdn.apple.com`) / IP (🇩🇪 eu1 + 🇷🇺 main) / порт (8444/8443/2053) чистым тестом (VPN off); подтверждено несколькими юзерами. Не зависит от IP/домена/порта — фингерпринт протокола.
  - Единственный непроверенный рычаг — порт 443 (fake-TLS сливается с HTTPS), но 443 занят REALITY на всех серверах (Мегафон/Yota при БС) — не роняем ради вторичной фичи.
  - eu1-прокси оставлен живым (домен `swcdn.apple.com`), `/proxy` отдаёт; в боте уже редирект на «Подключить VPN». Блоки РКН ротируются — **перепроверить, когда/если вернётся** (не по каждой жалобе). Детали: `SESSION_SUMMARY_2026-06-05.md`, память `project_vpn_mtproxy_rkn_block`.

- **Мегафон при белых списках** (выявлено 2026-05-22):
  - Yota и Мегафон имеют **разные whitelist** — несмотря на MVNO-связь. Наш main IP `81.200.146.32` (Timeweb) whitelisted у Yota, но **не whitelisted у Мегафон** — friend получает timeout (TCP-handshake даже не устанавливается).
  - Текущая main REALITY с SNI=cloud.mail.ru работает на Yota, не работает на Мегафон.
  - **Нужен:** разведка через Мегафон-друга при активных БС (серия IP-проб разных провайдеров для определения whitelist-friendly подсети). Перед разведкой убедиться что БС реально активны (контроль `digitalocean.com`).
  - Кандидаты VPS если Timeweb не пропускают: Mail.ru Cloud (mcs.mail.ru), VK Cloud, Selectel, Sber Cloud.
  - Архив гипотез про XHTTP: `docs/archive/yandex-cdn-xhttp-postmortem.md`.
  - Текущее состояние бота: кнопка Мегафон/Yota отдаёт VLESS+XHTTP CDN ссылку.

- **«Волна» оператор — обход белых списков** (новое 2026-05-28, обратная связь Unatham, Крым):
  - Юзер сообщил: на операторе «Волна» (Крым) обход БС не работает. Белые списки активны 3+ суток.
  - Контекст: «Волна» — крымский MVNO, whitelist у них вероятно отдельный (не как у Yota/Мегафон/main-операторов).
  - Owner предложил коллаб: продлеваем триал, юзер тестит наши решения.
  - **Нужен:** живой тест с устройством на «Волне». Подзадачи:
    1. Дать юзеру 1–2 vless:// каналы (main+yc) для теста — он напишет какой работает.
    2. Если оба не работают — пробовать другие SNI / другой relay (Yandex CDN?).
    3. Если найдём whitelist-friendly endpoint — добавить «Волна» как отдельный operator в боте (`mobile_op_volna`).
  - Это даёт нам канал в Крыму, который у конкурентов скорее всего не покрыт.

## Идеи / будущие направления (не трогать без явного решения владельца)

- **Relay через Yandex CDN** — единственный перспективный путь при whitelist. Сложно, риск нарушения ToS Яндекса. Требует разведки (какие IP реально в whitelist).

- **Своё клиентское приложение (Kronos VPN)** — стратегическая задача на горизонт **6–12 месяцев** (обсуждено 2026-05-29):
  - **Зачем:** массмаркет (HitVpn, AvoVPN, VPN Пельмень) делает свои клиенты — это позволяет (1) адаптивный split tunneling — приложение детектит БС и переключается между full/split автоматически; (2) auto-failover между протоколами (WG → VLESS → XHTTP); (3) auto-update конфигов и подсеток без участия юзера; (4) UX «одна кнопка → подключился» — наш текущий UX «получи .conf → импортируй в Amnezia → подключись» отсекает нетехнических юзеров.
  - **Реалистичный путь:** форк open-source AmneziaVPN (уже умеет AmneziaWG, VLESS, кросс-платформенный). Добавляем нашу логику адаптивного split + наш брендинг.
  - **Объём:** 2.5–3 месяца full-time для одного разработчика. Если делать по часу в неделю — ~год.
  - **Юридически сложно:** Apple App Store банит VPN от РФ-разработчиков (TestFlight / Enterprise account как обходной путь); Google Play мягче + есть APK-distribution; в РФ формально нужен реестр РКН для массового VPN-сервиса (для приватного «для друзей» — серая зона).
  - **Условия для запуска:** ~500+ платных юзеров (стоимость разработки оправдывается), стабильный продукт (не каждую неделю меняем протоколы), подтверждённая боль «нужен split tunneling» через 2-3+ фидбэка от юзеров (сейчас никто кроме владельца про это не говорил).
  - **Промежуточный split tunneling через статичные .conf** обсуждался и отвергнут как half-measure (либо рушит «работает при БС», либо требует двух конфигов на выбор — оба не «волшебно»).
