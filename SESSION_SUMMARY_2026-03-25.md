# Резюме сессии 2026-03-25 — recovery Telegram/VPN через веб

## Контекст

Пользователи сталкиваются с проблемой, когда Telegram перестаёт подключаться из-за блокировок/поломок прокси. По ТЗ нужно:

- иметь простой вход по `Telegram ID` на сайте;
- по кнопке “восстановить Telegram” добиться работоспособности Telegram;
- по кнопке “восстановить VPN (конфиг)” быстро выдать актуальный VPN-конфиг;
- вынести восстановление на отдельную страницу, чтобы мониторинг не мешал пользователям.

## Выполнено в репозитории (код и документация)

1. **Веб-панель мониторинга (`Projects/VPN/web`)**
   - Recovery UI вынесен из главной страницы `/` на отдельный маршрут `/recovery`.
   - Добавлены фронтенд-компоненты для двух кнопок:
     - “Восстановить Telegram”
     - “Восстановить VPN (конфиг)”

2. **Веб-bэкэнд endpoint’ы (`Projects/VPN/web/app.py`)**
   - `POST /api/recovery/telegram-proxy`:
     - принимает `telegram_id`;
     - проверяет пользователя по `bot/data/users.json`;
     - перезапускает docker-контейнер Telegram proxy (кандидаты: `mtproto-proxy`, `mtproxy-faketls`) по server_id (`main`/`eu1`) через существующую SSH-инфраструктуру.
   - `POST /api/recovery/vpn`:
     - принимает `telegram_id` и `android_safe`;
     - регенерирует/создаёт WireGuard/AmneziaWG peer и возвращает клиентский конфиг текстом (для скачивания).

3. **Скрипты фронтенда**
   - создан отдельный файл `web/static/recovery.js`, чтобы monitoring не зависел от recovery-разметки.

4. **Telegram-бот**
   - обновлён текст команды `/start` (и `help`) — добавлена строка со ссылкой на `/recovery`, чтобы пользователи знали куда перейти при проблемах с Telegram.
   - добавлена опциональная переменная в `bot/config.py` и `env_vars.example.txt`: `VPN_RECOVERY_URL` (дефолт на текущий URL `http://81.200.146.32:5001/recovery`).

5. **UX**
   - на странице восстановления удалена кнопка/ссылка “Назад к мониторингу”, чтобы пользователи не уходили обратно в мониторинг.

## Как это выглядит для пользователя

- Открывает мониторинг: `http://81.200.146.32:5001/` (только мониторинг).
- При проблемах с Telegram открывает recovery: `http://81.200.146.32:5001/recovery`
- Вводит свой `Telegram ID`.
- Нажимает:
  - “Восстановить Telegram” — после нажатия дождаться результата; в интерфейсе показывается JSON от backend (информация о перезапуске прокси).
  - “Восстановить VPN (конфиг)” — получает скачивание конфигурационного файла под WireGuard/AmneziaWG.

## Что проверить (ручные проверки, выполнено)

- `/recovery` открывается корректно и содержит обе кнопки.
- “Восстановить Telegram” возвращает JSON с подтверждением перезапуска (например, `container`: `mtproto-proxy` или `mtproxy-faketls`).
- “Восстановить VPN (конфиг)” скачивает файл вида `vpn_<telegram_id>_<server>_amneziawg.conf` (в зависимости от preferred_server_id).
- Мониторинг `/` без recovery-блоков.

## Результат

Сформирован единый простой сценарий восстановления:

- Telegram: через перезапуск docker proxy по `telegram_id` + проверка пользователя
- VPN: генерация/регенерация актуального конфиг-файла по `telegram_id`
