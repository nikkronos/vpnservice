# Предпрод-план коммерциализации VPN (2026-04)

## 1) Текущая инфраструктура (as-is)

### 1.1 Узлы и роли

- **Fornex (`185.21.8.91`)**
  - `vpn-bot.service` (Telegram-бот, control plane)
  - `vpn-web.service` (панель + recovery, порт `5001/tcp`)
  - eu1 AmneziaWG (Docker, `amnezia-awg2`, интерфейс `awg0`)
  - MTProxy Fake TLS (внешний порт `8444`, т.к. `443` занят Xray)
- **Timeweb (`81.200.146.32`)**
  - main WireGuard (`wg0`) для RU-подключений

### 1.2 Логические VPN-слоты

- `rus1` (дефолт RU) и `rus2` -> физическая нода `main`
- `eu1` (дефолт EU) и `eu2` -> физическая нода `eu1`
- Хранение пиров: ключи формата `{telegram_id}:{server_id}`.

### 1.3 Пользовательские каналы восстановления

- Telegram:
  - `/proxy` — актуальная `tg://proxy` ссылка
  - `/proxy_rotate` — ротация секрета (owner-only)
  - `/get_config`, `/regen`, `/server`, `/status`
- Web recovery:
  - `GET /api/recovery/proxy-link` — показать актуальную ссылку (без рестарта контейнера)
  - `POST /api/recovery/telegram-proxy` — рестарт контейнера + вернуть ссылку
  - `POST /api/recovery/vpn` — восстановление VPN-конфига для `eu1`/`eu2`

## 2) Главные риски перед платным запуском

## 2.1 Технические

1. **Single point of failure:** control plane (бот+веб) и eu1 сейчас на одном хосте Fornex.
2. **JSON как primary storage:** ограниченная транзакционность и аудит для биллинга/подписок.
3. **Web runtime:** Flask dev server не является production-стеком.
4. **Недостаточная observability:** нужны алерты, метрики, SLA-ориентированные проверки.
5. **Операционные секреты/ключи:** требуются регулярная ротация и формализация доступов.

## 2.2 Продуктовые

1. **Поддержка может не масштабироваться** без self-service и понятных сценариев fallback.
2. **Сетевые блокировки LTE/ASN** могут ломать отдельные входы (нужен портфель входов и сценарии переключения).
3. **Онбординг в оплату и продление** пока не автоматизированы.

## 2.3 Коммерческие/юридические

1. Публичный платный сервис требует оферту, privacy policy, abuse policy.
2. Платежи требуют корректной обработки webhook, статусов и возвратов.

---

## 3) Минимум, который обязателен до платного старта

1. **База данных (PostgreSQL)** вместо JSON как source of truth:
   - users, subscriptions, invoices, payments, devices, audit_log.
2. **Production web stack**:
   - `nginx + gunicorn` (или аналог), healthchecks, restart policy.
3. **Мониторинг и алертинг**:
   - uptime checks, error alerts, базовые метрики нагрузки и API.
4. **Бэкапы и восстановление**:
   - nightly backup БД + регулярная проверка restore.
5. **Безопасность платежей**:
   - webhook signature verify, idempotency keys, anti-replay.
6. **Подписочная логика**:
   - срок действия, grace period, блокировка/разблокировка доступа.
7. **ЛК/кабинет self-service (минимальный)**:
   - управление устройствами, регенерация конфига, история платежей.

## 4) Поэтапный план реализации

### Этап A: аудит и дизайн (1-2 недели)

- Формальный prelaunch-чеклист: `готово / риск / блокер`.
- Целевая архитектура БД и миграции с JSON.
- Модель подписок, статусов и доступов.

### Этап B: платформа (2-3 недели)

- PostgreSQL + слой доступа в коде.
- Миграция текущих пользователей/пиров.
- Перевод веба на production runtime.
- Мониторинг + алерты + backup restore test.

### Этап C: биллинг и коммуникации (2-4 недели)

- Интеграция оплаты (например Prodamus) и webhook flow.
- Email-канал как резерв (подтверждения, квитанции, напоминания).
- Автоматизация продления и окончания подписки.

### Этап D: закрытая бета

- 10-20 платящих пользователей.
- Замер SLA, нагрузки саппорта, конверсии оплат.
- Доработка UX до массового запуска.

## 5) Синхронизация с идеями из профильных чатов (AI/агенты/MCP)

Наблюдения из обсуждений ("агенты + MCP + Postgres + автоматизация рекламы") частично применимы к нашему проекту.

### 5.1 Что имеет смысл взять сейчас

1. **PostgreSQL как база знаний и транзакций** — полностью совпадает с потребностями биллинга и аудита.
2. **Agent-автоматизация рутин**:
   - подготовка отчетов по инцидентам,
   - синхронизация документации,
   - шаблоны ответов саппорта.
3. **MCP-интеграции** (но точечно):
   - платежные статусы,
   - почтовые сервисы,
   - сервис-деск/CRM (если появится).

### 5.2 Что не является приоритетом прямо сейчас

1. Агент "как Plurio для Meta Ads" — это другой продуктовый домен.
2. Meta Ads MCP не закрывает наши core-риски инфраструктуры VPN.

### 5.3 Практический компромисс

- Использовать AI-агентов для **ops и документации**, а не как ядро сервиса.
- Критический контур (доступы, платежи, выдача конфигов) должен быть детерминированным и тестируемым без "магии".

## 6) Definition of Ready перед монетизацией

Считаем сервис готовым к платному запуску, когда выполнены одновременно:

1. Нет блокирующих инцидентов по доступности за последние 14 дней.
2. БД/бэкапы/restore-проверки в штатном режиме.
3. Оплата -> активация/продление доступа работает автоматически и идемпотентно.
4. Recovery-сценарии работают без ручного участия владельца.
5. Юридические документы опубликованы и согласованы с моделью оплаты.
