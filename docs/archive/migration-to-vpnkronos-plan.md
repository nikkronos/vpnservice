# План переезда на @vpnkronos_bot

> Runbook для следующей сессии. Выполняется когда владелец говорит «делаем переезд».
> Связанные документы: `ROADMAP_VPN.md` (Фаза 3b proper), `docs/sessions/SESSION_SUMMARY_2026-05-27.md`.

> **Параметры утверждены владельцем 2026-05-27/28:**
> - День broadcast'а: **28.05 утром МСК**.
> - Владелец сам отправит broadcast через старый бот (`/broadcast` command).
> - Wait-window: **8 часов** (не 24).
> - **Enforcement включаем** в этой же операции.
> - Email-кампании — отдельная задача позже. На переезд берём только сбор email в БД.

---

## Цель

Перевести 34 текущих юзера со старого бота `@thisvpnforfriends_bot` на новый `@vpnkronos_bot`. Без потери данных. С обязательной email-верификацией для всех. Без миграции тех, кто не отреагирует — они дотлеют на старых конфигах ещё 24ч, потом конфиги сбрасываются и они вынуждены прийти в новый бот.

**Принципы:**
- Аддитивно: ничего из работающего не ломаем заранее.
- Селективный сброс: у тех, кто перешёл — всё работает; у тех, кто нет — отвалится через 24ч.
- БД одна (Fornex SQLite). Юзер идентифицируется по `telegram_id`, токен бота не имеет к нему отношения.
- Owner делает только два действия: broadcast и команду «переключаем». Остальное — Claude.

## Этапы

### Этап 0 — Pre-flight (Claude, ДО команды владельца)

Готовим всё, не трогая текущий прод.

#### 0.1. Миграция БД
- Колонка `users.migrated_at TEXT NULL` — отметка о первом `/start` в новом боте.
- Миграция идемпотентная, добавляется в `_migrate_add_migrated_at_column()` в `bot/database.py`.

#### 0.2. Хелперы БД
- `db_mark_migrated(telegram_id)` — выставляет `migrated_at = datetime('now')` если NULL (idempotent).
- `db_get_non_migrated_users()` — возвращает список юзеров с `migrated_at IS NULL AND active=1 AND telegram_id IS NOT NULL` для selective reset.

#### 0.3. FSM-онбординг в боте

Состояния (через `telebot.handler_backends.StateMemoryStorage`):

| Состояние | Что происходит |
|---|---|
| `AWAITING_GREETING_ACK` | Юзер видит приветствие, жмёт «Продолжить» |
| `AWAITING_EMAIL` | Юзер вводит email (если у него уже верифицирован — даём кнопки «Использовать этот / Ввести другой») |
| `AWAITING_OTP` | Бот шлёт OTP на email, юзер вводит код |
| `AWAITING_DISCLAIMER_ACK` | Показываем дисклеймер, юзер жмёт «Понятно» |
| (done) | Авто-trial если положен (`db_ensure_signup_trial`), переход в основное меню |

Логика входа в FSM:
- При `/start` в новом боте → `db_mark_migrated(tid)`.
- Если `db_get_subscription(tid)` показывает `email_verified` и `migrated_at_just_set=False` (был миграция раньше или это второй заход) — пропускаем FSM, сразу в меню.
- Иначе — запускаем FSM с шага 1.

Существующие 34 grandfather:
- `email_verified=1` если они когда-либо верифицировали email через ЛК. Иначе `0`.
- У большинства — `0` (не верифицировали). Они проходят полный онбординг.
- Их `expires_at IS NULL` (grandfather). Триал-кнопка не показывается (db_ensure_signup_trial — noop для них).

#### 0.4. Команда `/admin/migrate_reset` (только для owner_id)

Двушаговая для безопасности:
```
Owner: /migrate_reset
Bot: ⚠️ Это сбросит конфиги у N юзеров (migrated_at IS NULL).
     Список: @user1, @user2, ... (показать первые 10).
     Введи RESET чтобы подтвердить.
Owner: RESET
Bot: Сбрасываю... [прогресс]
Bot: Готово. Сброшено N конфигов.
```

Для каждого не-перешедшего юзера:
1. Удалить AmneziaWG peer:
   - `awg set awg0 peer <pubkey> remove` через `docker exec amnezia-awg2`
   - `/opt/amnezia-remove-client.sh <pubkey>`
   - `/opt/amnezia-save-conf.sh` (persist)
   - Очистить запись из `peers.json` и SQLite (если есть)
2. Удалить VLESS UUID:
   - `python3 /opt/xray-remove-client.py <uuid>` на каждом сервере (eu1, main, yc) где он был
   - Очистить `vless_uuid` в БД
3. Регенерировать `sub_token`:
   - `UPDATE users SET sub_token = NULL` → следующий `db_ensure_sub_token` сгенерит новый
   - У старого `sub_token` уже не будет соответствия → старая subscription-URL вернёт 404/empty
4. **НЕ удаляем** запись юзера и email — они могут вернуться через новый бот.

Логирование каждой операции (в `journalctl` + сводный отчёт владельцу).

#### 0.5. Тесты pre-flight (на текущем боте)
- Тест FSM-онбординга на тестовом telegram_id (например, мой бот-тестировщик или second-account владельца).
- Тест `/admin/migrate_reset` в dry-run режиме (флаг `--dry-run`?) — показывает что будет сброшено, но не сбрасывает.

#### 0.6. Деплой кода
Все pre-flight изменения деплоятся в текущий бот `@thisvpnforfriends_bot`. Они **аддитивны**:
- Колонка `migrated_at` ни на что не влияет пока миграция не запущена.
- FSM-онбординг срабатывает только если `migrated_at IS NULL` и юзер впервые /start. Существующие 34 юзера сейчас все имеют `migrated_at = NULL`, но они **уже /start'нули** старого бота → их БД-запись имеет другие маркеры (например, `email_verified` или AWG-peer). Нужно гарантировать что мы не запустим онбординг для них СЕЙЧАС, до переезда.
- Решение: FSM включается только если установлен ENV-флаг `ONBOARDING_ENABLED=1`. До переезда — `0`, FSM спит. После swap токена — выставляем `1`.

### Этап 1 — Broadcast (Owner)

Owner делает в текущем `@thisvpnforfriends_bot`:
```
/broadcast <текст>
```

**Финальный текст broadcast (утверждён 2026-05-27):**

```
Для поддержания и улучшение сервиса в рабочем виде переезжаем на новый бот.

В течение суток данный бот и vpn-конфиги перестанут работать.

Ссылка на новый бот: @vpnkronos_bot
Ссылка на новостной тг канал: https://t.me/vpnkronos
Ссылка на сайт с личным кабинетом: https://supportkronos.online:8443/recovery (будет работать в обход белых списков)
```

Примечание для исполнения: фактический срок до сброса — **8 часов** (но в тексте «в течение суток» — оставлено как мягкий буфер для юзеров).

**После broadcast:** засекаем 24 часа. В это время:
- Часть юзеров переходит → они /start'нут новый бот → им включится FSM (после swap)... подожди, FSM пока не включён.

Хмм. Конфликт. Нужно подумать.

**Решение:** broadcast делается ПОСЛЕ swap токена. Тогда:
- Старый бот молчит (нет кода).
- Новый бот живой с включённым онбордингом.
- Broadcast рассылается через TG API напрямую (не через бота, а через `getUpdates`/`sendMessage` от старого токена) — последняя возможность достучаться до юзеров со старого канала.

**Но** TG-API для broadcast не существует — sendMessage можно слать только если у бота есть chat_id юзера. У старого бота он есть (юзеры с ним переписывались). У нового — нет (пока не /start'нут).

**Финальный план:** broadcast делается ДО swap токена. Но онбординг включается только после swap (через ENV-флаг). А юзеры, которые /start'нут новый бот ДО swap — увидят «бот не настроен» (т.к. там старый код ещё). Они попробуют ещё раз позже.

**Альтернатива (чище):** broadcast делается **из старого бота** прямо перед swap. И сразу после broadcast — swap. У юзеров есть несколько секунд/минут где старый бот молчит, новый ещё не настроен. Это ОК если они не пытаются /start именно в этот момент.

### Этап 2 — Token Swap (Claude, по команде владельца)

После broadcast (или одновременно — мы перешлём текст через старый бот, потом сразу свапаем).

#### 2.1. Backup
```bash
ssh fornex "cp /opt/vpnservice/env_vars.txt /opt/vpnservice/env_vars.txt.bak.$(date +%s)"
```

#### 2.2. Swap токена
- Получить от owner: текущий токен `@vpnkronos_bot` (формат: `8828336517:AAE3PIry...`).
- Заменить `BOT_TOKEN=...` в `/opt/vpnservice/env_vars.txt`.
- Поставить `ONBOARDING_ENABLED=1` (если ещё нет).

#### 2.3. Рестарт
```bash
ssh fornex "systemctl restart vpn-bot.service && sleep 2 && systemctl is-active vpn-bot.service"
ssh fornex "systemctl restart vpn-web.service"  # web использует BOT_TOKEN для уведомлений + Stars
```

#### 2.4. Smoke-тест
- Со второго аккаунта владельца: `/start` в `@vpnkronos_bot` → должен запуститься онбординг.
- Через Mini App-кнопку: проверить auto-login.
- Проверить Stars-кнопки, donation-flow.

#### 2.5. Старый бот
`@thisvpnforfriends_bot` больше не отвечает на сообщения (его токен больше нигде не используется). Юзеры, которые туда напишут — увидят, что бот не отвечает. Это нормально и соответствует broadcast'у.

### Этап 3 — Ожидание 8 часов

В это время:
- Часть юзеров /start'нут `@vpnkronos_bot` → пройдут онбординг → `migrated_at` выставится → их конфиги останутся работать.
- Часть не отреагирует → `migrated_at` останется `NULL`.

**Мониторинг (Claude каждые несколько часов):**
```bash
ssh fornex "/opt/vpnservice/venv/bin/python -c '
from bot.database import _conn, init_db
init_db()
with _conn() as con:
    total = con.execute(\"SELECT COUNT(*) FROM users WHERE active=1 AND telegram_id IS NOT NULL\").fetchone()[0]
    migrated = con.execute(\"SELECT COUNT(*) FROM users WHERE active=1 AND telegram_id IS NOT NULL AND migrated_at IS NOT NULL\").fetchone()[0]
    print(f\"Migrated: {migrated}/{total}\")
'"
```

### Этап 4 — Selective Reset (через +8 часов)

Owner вызывает в `@vpnkronos_bot`: `/migrate_reset` → `RESET`.

Бот сбрасывает конфиги у не-перешедших. Отчёт владельцу:
```
✅ Сброшено: 8 юзеров.
   AWG peers удалены: 8
   VLESS UUIDs удалены: 5 (3 без VLESS)
   sub_tokens регенерированы: 8

Список не-перешедших: @user1, @user2, ...
```

После этого их устройства теряют доступ → они приходят в новый бот → проходят онбординг → получают новые конфиги.

## Rollback (если катастрофа)

Если после swap токена что-то идёт не так — откат:
```bash
ssh fornex "cp /opt/vpnservice/env_vars.txt.bak.<timestamp> /opt/vpnservice/env_vars.txt"
ssh fornex "systemctl restart vpn-bot.service vpn-web.service"
```

Через 2 минуты `@thisvpnforfriends_bot` снова отвечает, юзеры в том же состоянии что до swap. БД не пострадает (мы ничего не удаляли).

`/admin/migrate_reset` — необратима в части AWG peers (удалённый peer не вернёшь). НО peers генерируются заново при `/get_config` — юзер просто получит новый. Только тот, кто **уже** не зайдёт в новый бот, останется без VPN — что и было задачей.

## Длительность

- Pre-flight (Claude pre-work): ~3-4 часа (онбординг FSM + команда reset + тесты).
- Execution day: ~30 минут (swap + verify).
- 24h wait: автоматический.
- Reset: ~5 минут.
- Verify: ~30 минут.

**Итого активной работы:** ~5-6 часов в одну сессию (включая ожидание broadcast'а).
