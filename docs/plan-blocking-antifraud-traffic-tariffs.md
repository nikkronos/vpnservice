## 1. Блокировки РКН — резильентность

**Диагноз (durable, см. `project_vpn_rkn_ip_block`):** РКН режет **по IP-репутации, НЕ по транспорту.** На фильтрующих сетях (T2-тип) иностранные IP (Fornex) и VPN-хостинги (Timeweb) блокируются; **крупное РФ-облако (Яндекс) проходит.** Транспорт tcp↔xhttp не влияет (доказано: eu1 и yc идентичны кроме IP — eu1 не работает, yc работает). Мегафон-БС: не пускает ни один облачный IP, только домены — тяжёлый гэп.

**Сделано:** yc2 (2-й Яндекс REALITY, клон yc) — резерв + разгрузка yc для filtered-сетей.

**План:**
- [ ] **Перенести subscription-fetch URL** с `supportkronos.online:8443` (немецкий IP) на РФ-IP (Яндекс) — сейчас под БС подписка не обновляется (HAPP берёт кэш). Reverse-proxy `/sub` на yc/yc2.
- [ ] **Ещё Яндекс-эндпоинты** по мере роста нагрузки/блокировок IP (Яндекс — выживший; другие РФ-облака VK/Mail/Selectel под Мегафон-БС НЕ прошли).
- [ ] **Сигнал блокировки = жалобы юзеров по оператору** (не health-check — он видит только «сервер поднят»). Завести лёгкий канал сбора (support).
- **НЕ делать:** гнаться за транспортом (gRPC и т.п.) для решения IP-блока — не поможет. Не менять проверенные каналы по непроверенной теории (урок 06-06: дважды зря гонялись за транспортом).
- **Мегафон-БС** — отложено (нужен whitelist-friendly эндпоинт, пока не найден).

## 2. Анти-фрод (шеринг ссылок)

**Контекст:** юзеры шерят ссылки (подтверждено фидбэком). Нужно (а) видеть шеринг, (б) ограничивать по тарифу.

**Инструмент (готов):** `scripts/access_audit.py` — энумератор всех кредов на всех серверах + инвариант «кред ↔ активный юзер И отзываем». Текущий прогон: единственная untracked-зона = **eu1 shared-пул (11 UUID)**; main/yc/yc2 — 100% per-user.

**Фрод-зона eu1 (закрытие):** ⛔ **НЕ ЗАКРЫВАТЬ `--no-shared`** — вскрыто 2026-06-10. Попытка Этапа 4 (`sync_eu1_vless.py --no-shared`) **перерубила выход «Европы»**: eu1 vless-ws — приёмник релея **yc/yc2 → eu1** (UUID `359e23cc`, не фрод, а инфра). Откатили из бэкапа. Подробно: `DONE_LIST_VPN.md` (2026-06-10), память `project_vpn_yc_relay_topology`. Этапы 0-2 (per-user backfill на ПРЯМЫХ входах vless-tcp/xhttp) — остаются в силе (grace-режим). Защита: `sync_eu1_vless.py` теперь бережёт `RELAY_PRESERVE` и гейтит `--no-shared` за `--force`. Broadcast НЕ нужен. Переразбор «11 shared»: 1 (vless-ws) = релей; 9 (vless-tcp) — статус не доказан, поштучно.

**Лимит устройств (ключевая анти-фрод задача, НЕ начата):**
- [ ] **Per-device слоты** вместо per-platform (сейчас ключ = `{tid}:{server}:{platform}`, iPad+iPhone = один `ios`). Нужны именованные устройства.
- [ ] **Xray iplimit** — ограничение числа одновременных IP на UUID. У **3x-ui это встроено** («Лимит IP» + «Лог IP-адресов») — взять оттуда механизм. Это и есть enforcement лимита устройств.
- Это **keystone-задача**: разблокирует и тарифы (семейный/соло), и анти-фрод.
