## 0. Факты инфраструктуры (проверено на проде 2026-06-10)

- **peers** PK = `(telegram_id, server_id, platform)`, `platform ∈ {pc,ios,android}` (`bot/database.py`). Коллизия: iPad+iPhone = оба `ios` → один слот (фидбэк Ани).
- Все peer-функции (`create/regen/delete/find_peer_by_telegram_id`) параметризованы `platform` (`bot/wireguard_peers.py`).
- VLESS: per-user UUID на сервер (`vless_uuid_main/yc/eu1`; yc2 = клон yc). Один UUID юзера общий для всех его устройств.
- **Access-log (loglevel info) на ВСЕХ 4 входах пишет реальный client-IP + `email: tid_X@kronos`:**
  - eu1-direct (`[vless-tcp >> direct]`), main (`[vless-tcp >> direct]`), yc/yc2 (`[vless-xhttp -> eu1]` — релей).
  - **Релей-нюанс:** реальный IP клиента виден на ВХОДЕ (yc/yc2/main/eu1-tcp). На eu1-приёмнике релея — только `from 127.0.0.1` (бесполезно для per-user). Значит парсить надо логи входов.
- Xray **не имеет** нативного per-user IP-лимита → строим как у 3x-ui: парсинг access-log + enforcement.

## 1. Два механизма — что каждый реально решает (важно не путать)

| Механизм | Что реально enforce'ит | Не enforce'ит |
|---|---|---|
| **iplimit** (distinct IP/окно на `tid`) | анти-шеринг: разные люди/локации на одном аккаунте | «N устройств на одной Wi-Fi» (семья за одним IP = 1 IP при любом числе устройств) |
| **per-device слоты/креды** | точный счёт устройств (каждое = свой кред) | сам по себе шеринг между локациями (если юзер раздал свои per-device креды) |

**Вывод:** «лимит устройств» в тарифе на практике = **iplimit-порог** (distinct IP), а не буквальный счёт девайсов. Это и есть рыночная норма (конкурент «2 устройства» = мягкий анти-шеринг). Буквальный per-device-счёт для VLESS (per-device UUID) — тяжелее и не нужен на старте. **per-device СЛОТЫ нужны отдельно — для AWG** (фикс коллизии iPad/iPhone + «семейный»), это UX/инфра, не enforcement.

Итог: **iplimit = анти-фрод + основа тарифных лимитов; per-device слоты = UX-фикс AWG + мультидевайс.** Оба нужны, но независимы.

## ЧАСТЬ A — iplimit (VLESS, анти-фрод + лимит)

### A1. Архитектура (распределённый, под наш релей)

```
[eu1-tcp log] [main log] [yc log] [yc2 log]   ← real IP + email на входах
      │           │         │        │
      └───────────┴────┬────┴────────┘
         per-server watcher (cron): парсит свежий access-log,
         собирает {tid → set(IP) за окно W} → шлёт на Fornex
                          │
              central: таблица ip_usage(tid, ip, last_seen, server)
                          │
              aggregate: distinct_ip(tid) за окно W по всем входам
                          │
        ┌─────────────────┴─────────────────┐
   Stage 1 (observe)                   Stage 2 (enforce)
   видно в админке + алерт              действие при превышении
   при превышении, НЕ рубим             (уведомление/мягкий ребан)
```

- **Сбор IP:** watcher на каждом входе (eu1 локально; main/yc/yc2 через SSH, как health_check батчем) парсит `journalctl -u xray` за последнее окно, regex `from (IP):port accepted ... email: tid_(\d+)@kronos`, отдаёт `{tid: {ip...}}`. eu1 — только `vless-tcp` (direct); vless-ws/xhttp на eu1 = релей (127.0.0.1) — скипать.
- **Хранение:** таблица `ip_usage(telegram_id, ip, server_id, first_seen, last_seen)` на Fornex (upsert по (tid,ip)). Retention: чистим записи старше окна (напр. 24ч).
- **Агрегация:** `distinct IP за окно W` per tid (W — параметр, см. A2).
- **Cron:** `*/5` watcher (как traffic_accounting). Лёгкий (парсинг текста).

### A2. Порог и окно — нюансы (нужно решение владельца)

- **Окно W:** «одновременность» из логов точно не достаётся → берём **rolling distinct-IP за W минут**. Кандидат W = 10–15 мин (живая одновременность), плюс отдельный суточный счётчик для «широкого шеринга».
- **Мобильные IP меняются** (4G handoff, смена БС): легитимный юзер может дать 2–3 IP за час САМ. Поэтому порог с запасом: тариф «3 устройства» → алерт-порог напр. **≥4–5 distinct IP за 15 мин** (а не ровно 3). Цель — ловить продажу/раздачу на 10+ человек, не штрафовать честного.
- **CGNAT/общий IP** (несколько юзеров за одним провайдерским IP): редко мешает (мы считаем IP на tid, не tid на IP).
- **IPv6:** если клиенты по v6 — считать /64-префикс, не полный адрес (один девайс меняет v6 в пределах /64).

### A3. Stage 1 — ТОЛЬКО наблюдение (рекомендую начать с него)

Замерить РЕАЛЬНОЕ распределение distinct-IP per юзер ПРЕЖДЕ чем ставить пороги/рубить (урок инцидента: не на глаз). Дельта: `scripts/ip_usage_watcher.py` (cron `*/5`) + таблица + карточка в админке «distinct IP за 15м/24ч на юзера» + TG-алерт при явном превышении (напр. >8 IP/15м). **Никакого авто-ребана.** За неделю наблюдения — увидим, где реальная граница «честный multi-network» vs «шеринг», и откалибруем пороги по данным.

### A4. Stage 2 — enforcement (после калибровки)

Варианты действия при подтверждённом превышении (выбрать позже, по данным Stage 1):
- (a) уведомление юзеру «слишком много одновременных подключений — это против правил, апгрейдни тариф/перестань шерить»;
- (b) мягкий ребан: временно убрать per-user UUID из конфигов (через механизм `sync_xray_users`, как enforce_expired) на cooldown;
- (c) троттлинг.
Рекомендация: (a) сначала, (b) для злостных. **Обратимо, с уведомлением — не рубим вслепую.**

### A5. Риски A
- Парсинг journald: при ротации/высокой нагрузке окно может «убегать» — watcher с overlap (как share-audit sampler).
- Ложные срабатывания на мобильных — лечится запасом порога + Stage 1 калибровкой.
- Доп. SSH-нагрузка (3 входа /5 мин) — батчить, как health_check.

## ЧАСТЬ B — per-device слоты (AWG, UX + мультидевайс)

### B1. Проблема
PK `(tid, server, platform)`, `platform ∈ {pc,ios,android}` → 2 девайса одной ОС коллизят; «обновить конфиг» на одном ломает второй (фидбэк Ани).

### B2. Целевая модель
Ключ слота → **`(tid, server, device_id)`**, где `device_id` — стабильный идентификатор именованного устройства (не ОС). Имя устройства — человекочитаемое («iPhone Ани», «Рабочий ПК»).

Два варианта хранения (решение в доке-2 при реализации):
- **B2a (минимально):** заменить колонку `platform` → `device_id TEXT` в PK `peers`; имя+метаданные устройства — там же (`device_name`, `os`). Меньше таблиц.
- **B2b (чисто):** отдельная таблица `devices(device_id PK, telegram_id, name, os, created_at)` + `peers.device_id` FK. Чище под «список устройств/лимит», но больше рефактора.

### B3. Что затрагивается (объём рефактора)
- **Схема + миграция:** PK `peers`; existing peers → device_id (pc→«ПК», ios→«Устройство iOS», android→«Android-устройство»; имя юзер потом переименует). Идемпотентная миграция + бэкап (есть `migrate_peers_check.py`, `test_peers_sqlite.py` — расширить).
- **`bot/wireguard_peers.py`:** `platform` → `device_id` во всех функциях (create/regen/delete/find). ~10 сигнатур.
- **`bot/storage.py`:** нормализация ключа.
- **UX бота:** вместо «выбери платформу» — «список устройств» + «добавить устройство (имя)» + «обновить КОНКРЕТНОЕ устройство» + «удалить устройство». Закрывает ловушку «обновил — сломал соседнее».
- **ЛК (`recovery.js`/`app.py`):** управление устройствами (список/имя/QR на устройство).
- **AmneziaWG:** каждое устройство = свой peer (свой keypair/IP/conf) — это и так уже так по слотам, просто слот теперь именованный, а не per-OS.

### B4. Риски B
- Большой рефактор рабочего peer-флоу → **поэтапно**, с тестами (`test_peers_sqlite.py`) на временной БД ПЕРЕД продом; миграция с бэкапом и сверкой (`peers_sync_check.py`).
- VLESS per-device НЕ делаем здесь (VLESS остаётся per-user + iplimit) — per-device касается только AWG-слотов. Это сознательно (Оккам).
