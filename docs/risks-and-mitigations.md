# Риски инфраструктуры VPN — реестр

**Создано:** 2026-05-07  
**Статус:** активный документ, обновлять по мере изменений

---

## 🔴 Критические — сломают VPN полностью

### R1. eu1 (Fornex) недоступен
**Описание:** Вся цепочка `телефон → YC VM → eu1 → интернет` рвётся на последнем звене. YC VM без eu1 как outbound бесполезен.  
**Вероятность:** Средняя (хостинг может упасть, Fornex может заблокировать сервер)  
**Митигация:**
- Мониторинг eu1 (веб-панель `http://185.21.8.91:5001/`)
- В конфиге YC VM есть `"direct"` outbound — при падении eu1 быстро переключить routing на `freedom` (выход напрямую из России)
- Команда для переключения: обновить конфиг YC через Python, `systemctl restart xray`

**Статус:** ⏳ Не реализована автоматическая fallback-логика

---

### R2. Xray на YC VM упал / конфиг сломан
**Описание:** Xray падает — `/mobile_vpn` перестаёт работать полностью.  
**Вероятность:** Низкая (systemd перезапускает автоматически)  
**Митигация:**
- `systemctl enable xray` — уже включено
- При сломанном конфиге: войти через SSH, починить через Python (не heredoc/sed — markdown-артефакты)
- Рабочий конфиг задокументирован в `docs/yandex-cloud-reality-setup.md`

**Статус:** ✅ Автозапуск настроен

## 🟡 Серьёзные — деградация сервиса

### R3. Yandex Cloud IP исключат из whitelist
**Описание:** Если whitelist-режим станет ещё жёстче, операторы могут исключить конкретные подсети YC.  
**Вероятность:** Низкая (блокировка YC = проблемы для тысяч российских сервисов)  
**Митигация:**
- Резервный вариант: Selectel, МТС Cloud, Sber Cloud — аналогичные российские провайдеры
- При следующем whitelist-событии проверить: если YC не работает — пробовать другие российские облака

**Статус операторов (тест 2026-05-07):**
| Оператор | Статус |
|---|---|
| T2 (Tele2) | ✅ Работает |
| МТС | ✅ Работает (Happ, подтверждено Риной) |
| Yota (MegaFon) | ❓ Не работало — возможно whitelist другой |
| Билайн | ❓ Не проверено |

**Статус:** ⏳ Резервный провайдер не подготовлен

---

### R4. Грант Yandex Cloud закончится
**Описание:** Грант 4000₽ при расходе ~1340₽/мес закончится через ~3 месяца (≈август 2026). После этого списывается с карты.  
**Вероятность:** Высокая (гарантированно, если не выключить VM)  
**Митигация:**
- Поставить бюджетный алерт в YC Console (Биллинг → Бюджеты)
- Если whitelist-режим не повторяется — рассмотреть удаление VM до исчерпания гранта
- Альтернатива: прерываемая VM (~400₽/мес вместо 1340₽) — опция не нашлась при создании, проверить позже

**Статус:** ⚠️ Алерт не настроен — **сделать**

---

### R5. Двойной хоп увеличивает задержку
**Описание:** Телефон → Москва (YC) → Европа (eu1) → интернет. Пинг выше, чем при прямом AmneziaWG.  
**Вероятность:** 100% (архитектурная особенность)  
**Влияние:** Некритично для браузера/мессенджеров, заметно для видеозвонков и игр  
**Митигация:**
- Использовать YC-Reality только когда AmneziaWG недоступен (так и задумано)
- При необходимости: добавить routing — российские домены через `freedom` (прямо из YC), зарубежные через eu1

**Статус:** ℹ️ Принято как допустимо

---

### R6. Xray на eu1 упал (VLESS+WS порт 80)
**Описание:** YC VM пытается подключиться к eu1:80 и падает — трафик не проходит.  
**Вероятность:** Низкая  
**Митигация:**
- Проверить: `journalctl -u xray -f` на eu1 (`185.21.8.91`)
- Перезапустить: `systemctl restart xray` на eu1

**Статус:** ⏳ Нет мониторинга

## 🟢 Управляемые — решаемы при возникновении

### R7. SSH ключ к YC VM потерян
**Описание:** Файл `C:\Users\krono\.ssh\id_ed25519` утерян — SSH-доступ к YC VM невозможен.  
**Вероятность:** Низкая  
**Митигация:**
- Войти через серийную консоль в YC Console (VM → Серийная консоль)
- Добавить новый публичный ключ через метаданные VM в YC Console

**Статус:** ✅ Путь восстановления известен

---

### R8. Приватный ключ Xray REALITY утерян (при пересоздании VM)
**Описание:** При случайном удалении VM ключи теряются — нужно перегенерировать и обновить ссылку в боте.  
**Вероятность:** Низкая  
**Митигация:**
- Приватный ключ задокументирован в `docs/yandex-cloud-reality-setup.md`
- При пересоздании: `xray x25519` → новые ключи → Python-запись конфига → обновить `VLESS_REALITY_SHARE_URL` в env_vars.txt на Fornex → перезапустить бота

**Статус:** ✅ Процедура восстановления задокументирована

---

### R9. Оператор замечает аномалию трафика (продвинутый DPI)
**Описание:** SNI = `www.yandex.ru`, но трафик идёт на IP Yandex Cloud (другие подсети). Теоретически детектируемо.  
**Вероятность:** Очень низкая (whitelist-режим — это грубый IP-фильтр, не DPI)  
**Митигация:**
- Сменить SNI на домен, реально размещённый на Yandex Cloud IP
- Или переключиться на другой российский облачный провайдер

**Статус:** ℹ️ Не актуально при текущих условиях

---

### R10. Markdown-артефакты при редактировании конфига
**Описание:** При копировании URL из chat-интерфейса `www.yandex.ru` превращается в `[www.yandex.ru](https://...)` в файлах и конфигах.  
**Вероятность:** Высокая (уже происходило дважды)  
**Митигация:**
- Всегда использовать Python для записи JSON конфигов и env_vars.txt
- Проверять: `grep -o 'sni=[^&]*' /usr/local/etc/xray/config.json` после изменений

**Статус:** ✅ Известная проблема, задокументирована
