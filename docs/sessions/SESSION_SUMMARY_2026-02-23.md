4. **Переустановка MTProto-прокси для Telegram на eu1**
   - На eu1 заново развёрнут MTProto‑прокси в Docker:
     ```bash
     sudo docker run -d --name mtproto-proxy --restart=always -p 443:443 telegrammessenger/proxy
     sudo docker logs mtproto-proxy 2>&1 | grep -o 'secret=[a-f0-9]*' | head -1
     ```
   - Получен новый `secret` и сформирована ссылка:
     ```
     tg://proxy?server=185.21.8.91&port=443&secret=9aea73d44c74a85d6045e41bf0aa9a0c
     ```
   - Прокси протестирован в Telegram‑клиенте — подключение успешно.

5. **Обновление ссылки MTProto в боте (Timeweb)**
   - На сервере Timeweb (`/opt/vpnservice`) обновлён файл `env_vars.txt`:

   - Бот перезапущен:
     ```bash

     systemctl restart vpn-bot.service
     systemctl status vpn-bot.service  # статус: active (running)
     ```
   - Теперь команда `/proxy` в боте отдаёт актуальную ссылку MTProto‑прокси на eu1.

6. **Эксперимент с Xray (VLESS) на eu1 — признан неудачным**
   - На eu1 установлен Xray (`xray-core`), бинарь и unit: `/usr/local/bin/xray`, конфиг `/usr/local/etc/xray/config.json`.
   - Настроен минимальный inbound VLESS/TCP без TLS:
     - порт `21017`, `listen: "0.0.0.0"`;
     - `protocol: "vless"`, один клиент с UUID `933f34d2-3035-4933-8827-2bda519f2dfc`;
     - `decryption: "none"`, `network: "tcp"`, `security: "none"`;
     - outbound `freedom`.
   - Xray‑служба успешно стартовала (`systemctl status xray` → `active (running)`), с самого eu1 `curl -I https://www.google.com` и `curl -I https://ifconfig.me` возвращают `HTTP/2 200` → выход в интернет есть.
   - На ПК (Windows) установлен v2rayN, импортирован `vless://` профиль:
     ```text
     vless://933f34d2-3035-4933-8827-2bda519f2dfc@185.21.8.91:21017?encryption=none&security=none&type=tcp#eu1-xray-test
     ```
     Профиль подключается, в логах Xray и v2rayN видны `accepted tcp... [socks -> proxy]`, то есть туннель до eu1 рабочий.
   - Однако при включённом системном прокси/TUN в v2rayN:
     - `ifconfig.me` на ПК по‑прежнему показывает исходный российский IP;
     - заблокированные РКН сайты не открываются;
     - прямой запрос `curl.exe --proxy socks5://127.0.0.1:10808 https://ifconfig.me` в PowerShell не возвращает результата.
   - Вывод: на текущей конфигурации (VLESS/TCP без TLS/Reality) **практического выигрыша по обходу блокировок не достигли**, поведение клиентов остаётся нестабильным. Эксперимент с Xray в таком виде зафиксирован как неудачный; дальнейшее углубление (Flow/Reality, сложные режимы маршрутизации) сознательно **остановлено**, чтобы не тратить время и ресурсы без гарантии результата.

7. **Эксперимент с MTProto — блокировка оператором**

   - Контейнер `mtproto-proxy` работает, порт слушается (`docker-proxy`).
   - **Вывод:** мобильный оператор блокирует MTProto по сигнатуре протокола, а не по порту. MTProto на eu1 **не работает** для клиентов из России.
