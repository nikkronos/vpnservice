## Что сделано

### 1. Закрытие веб-панели от посторонних

**Проблема:** `http://185.21.8.91:5001/` была публично доступна — таблица пользователей
(usernames, IP, трафик) видна без авторизации.

**Итерация 1:** Basic Auth через HTTP — браузер не показывал диалог (Chrome подавляет
для HTTP), показывал «Access denied». Неудобно.

**Итерация 2:** убрали auth с API-роутов (JS не передаёт заголовки), оставили только на `/`.
Не сработало — браузер помнил credentials из URL и открывал без запроса.

**Итерация 3 (финал):** HTML-форма входа с Flask session:
- Создан `web/templates/login.html` — тёмная тема, минималистичная форма
- `app.py`: добавлены `/login` (GET/POST), `/logout`, `_require_admin_auth` проверяет `session["logged_in"]`
- `app.secret_key` = `ADMIN_SECRET` (сессии переживают перезапуск)
- Логин: `admin`, пароль: `ADMIN_SECRET` из `env_vars.txt`
- `/recovery` остался публичным
- API-роуты (`/api/traffic` и др.) — без auth (вызываются JS на странице)

### 2. Колонка «Прокси» в таблице пользователей

**Задача:** понять, использует ли пользователь MTProxy Telegram.

**Техническая реализация:**
- `bot/database.py`: миграция `proxy_requested_at TEXT` в таблицу `users` + функция `db_update_proxy_requested_at(telegram_id)`
- `bot/main.py`: вызов `db_update_proxy_requested_at` при отправке прокси-ссылки (кнопка «📎 Прокси Telegram» и команда `/proxy`)
- `web/app.py`: `api_traffic()` подгружает `proxy_requested_at` из БД и включает в ответ
- `web/static/main.js`: колонка «Прокси» с датой последнего запроса или «—»; добавлена функция `relDate()`
- `web/templates/index.html` (на сервере): добавлен `<th>Прокси</th>`

**Результат:** в панели видно, кто и когда запрашивал прокси-ссылку. Исторические данные
не восстановить — фиксируется с момента деплоя.

## Открытые вопросы
