### 2. Аудит текущего XHTTP relay через Yandex CDN

**Серверная сторона — норма:**
- nginx на Fornex корректно роутит: `Upgrade: websocket` → 8080 (WS), всё остальное → 8081 (XHTTP).
- Xray конфиг (`network: xhttp, path: /vpn`) валиден.
- 19.05.2026 в 19:17–19:18 в логах nginx есть валидные XHTTP-запросы (`GET /vpn/<uuid>?x_padding=...`) от двух клиентских IP. Серверная цепочка CDN → Fornex → Xray механически проходит.

**Клиентская сторона — НЕ работает (тестировал владелец с другом):**
- Текущая VLESS+XHTTP ссылка через `cdn.vpnnkrns.ru` **не работает на Мегафон/Yota при активных БС**.
- Те же VPN-ссылки работают: на Wi-Fi, на других операторах (T2/MTC/Билайн через REALITY).
- Точные симптомы при последнем тесте не зафиксированы (timeout / connected-no-traffic / нестабильность — нужно повторить тест с детализацией).

**Гипотезы причины (любая или комбинация — не подтверждены):**
- Cloudflare DNS (grey cloud) перед Yandex CDN: при БС могут блокироваться Cloudflare DNS-серверы или CNAME-цепочка `cdn.vpnnkrns.ru` → `*.topology.gslb.yccdn.ru`.
- Yandex CDN IP в whitelist, но клиент не может зарезолвить домен — DNS-резолв ломается.
- Прозрачный прокси Мегафон/Yota перехватывает соединение (аналогичный паттерн уже встречали для `.ru` SNI в REALITY — см. `yandex-cloud-reality-setup.md`).
- TCP/TLS до CDN-edge блокируется на уровне ТСПУ.

**Что нужно для следующей диагностики (план для повторного теста с другом):**
Цель — получить точную точку обрыва. С устройства на Yota/Мегафон при активных БС, **без VPN**:
1. `nslookup cdn.vpnnkrns.ru 8.8.8.8` и `nslookup cdn.vpnnkrns.ru` (системный DNS) — резолвится ли домен и в какой IP.
2. `ping <получившийся IP>` (если резолвится) — отвечает ли.
3. `curl -v http://cdn.vpnnkrns.ru/vpn` — что отвечает (если на устройстве нет curl, открыть URL в браузере и записать поведение).
4. Попытка подключения через v2rayNG/Streisand — логи клиента (что именно фейлится).
По результатам — точная гипотеза, точное решение.

### 3. План Yandex API Gateway

Создан документ `docs/yandex-api-gateway-plan.md` — подробный план реализации:

- Архитектура: клиент → `*.apigw.yandexcloud.net` (HTTPS) → API Gateway → HTTP/WS → Fornex
- Преимущество: нативный домен Yandex Cloud, без посредников DNS и без Cloudflare-цепочки
- Шаги: создание API Gateway → проверка доступности на Yota без VPN при БС → адаптация nginx → формирование VLESS-ссылки → обновление кнопки в боте
- Риски: лимиты API Gateway, неподтверждённая поддержка WebSocket в transparent-режиме, latency, ToS YC
- Альтернативы при отказе: Cloud Functions, YC Application Load Balancer + Yandex CDN с тех. поддержкой

**Не реализовано** — ждём момента, когда будет возможность тестировать с устройства на Yota/Мегафон при активных БС.

## Сильная гипотеза — отсутствие TLS

Текущая `cdn.vpnnkrns.ru:80` — HTTP без шифрования. Прозрачный прокси Мегафон/Yota гипотетически вмешивается в HTTP-трафик и ломает XHTTP-контейнер VLESS. Косвенное подтверждение: VLESS+WS через Cloudflare CDN (HTTPS:443) работает на тех же операторах.

**Главный путь решения — апгрейд CDN на HTTPS.** Подробный пошаговый план — `docs/cdn-tls-upgrade-plan.md`.
