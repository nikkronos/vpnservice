# Spec 08: мульти-вход, резервы и (опционально) вторая нода

## Ограничение владельца (актуально)

**Второй VPS не планируется (нет бюджета).** Вся дальнейшая работа по этой спеке ведётся в рамках **уже оплаченных** серверов:

- **main** — Timeweb (`81.200.146.32`): WireGuard, бот, панель, MTProxy Fake TLS (если развёрнут).
- **eu1** — Fornex (`185.21.8.91`): AmneziaWG в Docker, опционально Xray REALITY (`/mobile_vpn`).

Ниже два сценария: **B — только старые серверы** (основной для проекта сейчас) и **A — отдельный VPS** (если когда‑нибудь появится бюджет; тогда смысл RU2/EU2 сохраняется).

---

## Честное ограничение «всё на старом»

Если у абонента на LTE/операторе зафиксирован **blackhole до конкретного публичного IP** (трафик **не доходит** до сервера), то:

- **дополнительный порт**, **другой протокол** или **доменное имя на тот же IP** **не помогут** — блок на уровне маршрута до этого адреса;
- помогает только **другой IP / другой ASN** (другой VPS, чужой VPN и т.п.).

Зато на тех же двух машинах можно дать пользователю **несколько независимых способов**, которые помогают, когда режут **протокол или порт**, а не весь IP:

| Уже есть (проверить по факту) | Где | Зачем |
|-------------------------------|-----|--------|
| WireGuard | main | простой VPN, РФ |
| AmneziaWG (UDP) | eu1 | обфусцированный туннель, EU |
| VLESS + REALITY (TCP), `/mobile_vpn` | eu1 (если поднят Xray) | запас, когда UDP/другое ведёт себя иначе на мобильной сети |
| MTProxy Fake TLS | main | Telegram без полного VPN |
| Домен вместо IP в `tg://proxy` | main, тот же IP | удобство и смена IP без смены «имени» в будущем (если IP сменится на том же хосте) |

Итого «как несколько кнопок» **без нового VPS**: пользователь перебирает **main WG ↔ eu1 AmneziaWG ↔ `/mobile_vpn` ↔ `/proxy`**, а не «Россия‑2 / Европа‑2» на другом железе.

## Сценарий B (без нового VPS) — цели и задачи

1. **Документация и бот:** явно описать в `/help` / `instruction` цепочку: «не работает A → попробуй B» (без обещания, что всегда сработает на любом операторе).
2. **REALITY на main (опционально):** если на Timeweb ещё **нет** TCP‑входа REALITY, а на eu1 IP «чёрный» с LTE — имеет смысл поднять Xray REALITY **на main** (другой публичный IP уже есть — это **тот же** VPS, не второй). Отдельная переменная в env, вторая ссылка в боте (например расширить `/mobile_vpn` двумя ссылками: EU + RU). Детали деплоя — по аналогии с `docs/xray-vless-reality-eu1-deploy.md`, адаптировать под main.
3. **Домен для MTProxy:** A‑запись на IP main; в ссылке `server=домен` — см. обсуждение в README/операторском гайде; секрет тот же.
4. **Не ломать** текущие `main` / `eu1`: новые сервисы — **дополнительные** порты и переменные; firewall и конфликты с Damir/другими сервисами на Timeweb проверять вручную.

## Сценарий A (отложено): отдельный VPS — RU2 / EU2

Имеет смысл **только** при появлении бюджета и необходимости обойти **blackhole до IP** текущих нод.

- **EU2** — новый VPS, **другой** хостер/ASN; стек как у eu1 (AmneziaWG + опционально Xray).
- **RU2** — второй WireGuard в РФ на другом провайдере.
- Требования к env, боту и `server_id` — как в прежней версии спеки (изоляция `WG_<ID>_*`, расширение `/server`).

Рекомендуемый порядок **когда появится VPS**: сначала проверка доступности IP с проблемной SIM, затем развёртывание и интеграция в бота.

## Проблема и контекст (общее)

- Блокировки **неравномерны** по операторам: у одного работает связка VPN+прокси, у другого — нет.
- Симптомы Telegram **«Обновление»** / **«Соединение»** и тайм‑ауты VPN согласуются с DPI/blackhole/блоком по сигнатуре; для точного вывода нужны оператор, Wi‑Fi vs LTE, скрин.

## Риски и митигация (сценарий B)

| Риск | Митигация |
|------|-----------|
| Порт 443 на main занят MTProxy / другим сервисом | Выбрать другой TCP‑порт для REALITY; указать в ссылке. |
| Один REALITY UUID на всех | Как в spec-07: пока общая ссылка для доверенного круга; ротация вручную. |
| Ожидание «как второй VPS» без закупки | Не обещать обход полного blackhole до IP текущих нод. |
