# Алгоритм «разблокировки» Telegram

Документ описывает план сохранения доступа к Telegram при блокировках и троттлинге со стороны РКН. Формулировки в бот пока не вносятся — только стратегия и шаги для владельца/агента.

## Текущая ситуация

- Telegram уже **замедляют** (скорость отправки режется); у пользователей (в т.ч. в Москве) плохо работает и по Wi‑Fi, и по мобильной сети.
- **EU-конфиг (AmneziaWG на eu1)** не особо справляется — скорость слабая (удобно для обхода блокировок сайтов, но для мессенджера часто избыточная задержка и нагрузка на канал).
- **Раньше использовали прокси** — работало лучше, но прокси перестал работать.
- **Испробованный сторонний прокси:** 79.132.138.66:9443 (тип MTProto, ключ в настройках) — на текущий момент **не работает** (подключение/скорость не устраивают). Не наш инфраструктура; зафиксировано для истории.
- Цель: иметь что-то **похожее на прокси** — быстрый и стабильный доступ именно к Telegram, без обязательного «весь трафик через VPN».

---

## Почему VPN для Telegram не идеален

- Весь трафик идёт через туннель: выше задержка, нагрузка на сервер и канал.
- DPI может выделять и замедлять VPN-трафик — троттлинг мессенджера через VPN сохраняется.
- Прокси **только для Telegram** даёт: меньшую задержку, меньше ресурсов, трафик точечный и при правильной маскировке менее заметен для DPI.

---

## Почему старый прокси перестал работать

В проекте на eu1 поднимался **обычный MTProto** (образ `telegrammessenger/proxy`). Его **блокировал мобильный оператор по сигнатуре протокола** — не по порту (пробовали 443 и 8443). То есть «просто MTProxy» без маскировки DPI распознаёт и режет.

Вывод: нужен **MTProxy с маскировкой** (Fake TLS), чтобы трафик выглядел как обычный HTTPS к легитимному домену.

---

## Алгоритм разблокировки (по шагам)

### Уровень 1: Быстрый вариант — свой MTProxy с Fake TLS

**Идея:** Поднять MTProxy с Fake TLS (например, `nineseconds/mtg`) на отдельном VPS. Для DPI трафик выглядит как HTTPS к выбранному домену (например, 1c.ru и т.п.), что сильно снижает риск блокировки по сигнатуре.

**Шаги:**

1. **Выбрать площадку для прокси:**
   - **Вариант A — VPS в России.** Плюс: фильтрация на границе РФ к зарубежному серверу не применяется к этому каналу; трафик «Абонент → VPS РФ → Telegram» часто стабильнее и быстрее по пингу. Минус: политика хостера по прокси/туннелям (нужно проверить ToS).
   - **Вариант B — VPS в EU (например, другой хостинг, не Fornex).** Плюс: уже есть опыт с eu1. Минус: трафик через границу РФ может снова попадать под троттлинг.

2. **Развернуть MTProxy с Fake TLS:**
   - Пошаговая инструкция: **docs/mtproxy-faketls-deploy.md** (команды под копипаст, проверка порта 443, генерация секрета, запуск контейнера `nineseconds/mtg:2`, сбор ссылки).
   - Использовать образ `nineseconds/mtg:2`; секрет с маскировкой под домен (например, 1c.ru); порт 443; ссылка `tg://proxy?server=<IP>&port=443&secret=<секрет>` — раздавать только себе/доверенным.

3. **Проверить:** подключение в Telegram через «Настройки → Данные и память → Использовать прокси» (или по ссылке). Проверить скорость отправки/получения.

4. **Задокументировать:** IP/порт, где развёрнут прокси, способ получения секрета и ссылки, команды запуска/перезапуска контейнера (см. примеры в разделе «Ссылки» ниже).

**Если оператор снова начнёт резать по сигнатуре** — следующий уровень: смена домена маскировки или перенос на другой VPS/провайдер по той же инструкции.

---

### Уровень 2: Резерв — VPN только для Telegram

Если прокси (в т.ч. с Fake TLS) перестанут работать или развёртывать их нецелесообразно:

- Пользователи продолжают заходить в Telegram **через VPN** (AmneziaWG eu1 или WireGuard main).
- В документации и (при необходимости позже) в боте явно указать: «При блокировке/замедлении Telegram — включите VPN и откройте Telegram».

Этот уровень уже доступен сегодня; он не решает проблему «слабая скорость через EU-конфиг», но даёт запасной вариант доступа.

---

### Уровень 3: Смена провайдера/локации прокси

Если текущий VPS для MTProxy начнут блокировать по IP или по политике:

- Иметь готовую инструкцию развёртывания (Docker, секрет Fake TLS, порт 443).
- Перенести контейнер на другой VPS (другой провайдер или другая страна) и выдать новую ссылку.
- Рассмотреть **непопулярные** провайдеры/локации — они реже попадают в первые волны блокировок (см. blocking-bypass-strategy.md).

---

## Что не делать (по текущему опыту проекта)

- **Не полагаться на обычный MTProto без Fake TLS** на eu1 — уже проверено: блокируется оператором по сигнатуре.
- **Не вносить в бот формулировки** по блокировке Telegram, пока не принято решение о выдаче ссылки на прокси через бота (команда `/proxy` удалена; при необходимости можно завести отдельную команду/флоу позже).

---

## Краткий чеклист для владельца

1. [ ] Выбрать VPS для MTProxy (РФ или EU, проверить ToS на прокси/туннели).
2. [ ] Развернуть MTProxy с Fake TLS (nineseconds/mtg, секрет с маскировкой под домен, порт 443).
3. [ ] Собрать ссылку `tg://proxy?server=...&port=443&secret=...`, протестировать на себе и при необходимости на друзьях.
4. [ ] Записать в документацию: хост, команды запуска, как получить ссылку (без публикации самой ссылки в репо).
5. [ ] При блокировке/троттлинге — сменить домен маскировки или перенести прокси на другой VPS по той же инструкции.
